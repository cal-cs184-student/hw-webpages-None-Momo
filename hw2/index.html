Section I: Bezier Curves and Surfaces

Part 1: Bezier curves with 1D de Casteljau subdivision

De Casteljau's algorithm is a recursive method for evaluating Bezier curves. Given \( n \) control points and a parameter \( t \in [0,1] \), each step performs linear interpolation between consecutive pairs of points:

\[ p_i' = (1 - t) \cdot p_i + t \cdot p_{i+1} \]

This reduces the number of points by one at each level. After \( n - 1 \) recursive steps, a single point remains — the point on the Bezier curve at parameter \( t \). By varying \( t \) from 0 to 1, we trace out the entire curve.

In my implementation, evaluateStep() takes the current set of control points and returns a new vector of intermediate points by linearly interpolating each adjacent pair using the parameter \( t \). The GUI calls this function repeatedly, stepping through each level of the recursion.

Part 2: Bezier surfaces with separable 1D de Casteljau

To extend de Casteljau's algorithm from curves to surfaces, we use a separable approach. A Bezier surface is defined by an \( n \times n \) grid of 3D control points and two parameters \( (u, v) \). The evaluation proceeds in two stages:

  1. For each row of control points, fully evaluate the 1D de Casteljau algorithm at parameter \( u \). This reduces each row to a single 3D point, producing \( n \) intermediate points.
  2. Then, treat these \( n \) intermediate points as a single Bezier curve and fully evaluate at parameter \( v \) to get the final surface point.

My implementation consists of three functions:

  • evaluateStep(): Performs one level of de Casteljau interpolation on a vector of 3D points, identical in logic to Part 1 but for Vector3D with an explicit parameter \( t \).
  • evaluate1D(): Repeatedly calls evaluateStep() until only one point remains, fully evaluating de Casteljau for a single row.
  • evaluate(u, v): Applies evaluate1D at parameter \( u \) to each row of the control point grid, then applies evaluate1D at parameter \( v \) to the resulting column of points.

Section II: Triangle Meshes and Half-Edge Data Structure

Part 3: Area-weighted vertex normals

To achieve smooth Phong shading, we need to compute a normal vector at each vertex. The approach used here is area-weighted vertex normals: for each vertex, we iterate over all adjacent faces, compute each face's normal via the cross product of two edge vectors, and sum them up. Since the magnitude of the cross product is proportional to twice the triangle's area, larger faces naturally contribute more to the final normal. Finally, we normalize the sum to obtain a unit normal vector.

In my implementation, I traverse the faces around a vertex using the half-edge data structure. Starting from vertex->halfedge(), I loop via h = h->twin()->next() until returning to the starting half-edge. For each non-boundary face, I extract the three vertex positions, compute cross(p1 - p0, p2 - p0), and accumulate the result. The final normal is returned as n.unit().

Part 4: Edge flip

The edge flip operation takes a shared edge between two triangles and rotates it to connect the other pair of vertices. Given two triangles \( (a, b, c) \) and \( (c, b, d) \) sharing edge \( (b, c) \), flipping produces triangles \( (a, d, c) \) and \( (d, a, b) \), where the edge now connects \( (a, d) \).

My implementation first checks if the edge is on the boundary (in which case we skip the flip). Then I collect all 6 half-edges, 4 vertices, 2 faces, and the edge itself. After the flip, I reassign all the pointers using setNeighbors() for every half-edge, and update the halfedge() pointers for vertices, faces, and the edge. No new mesh elements are created or destroyed — only pointer reassignments.

Debugging journey: The key challenge was carefully tracking which half-edge belongs to which face after the flip. I drew out the before/after diagrams on paper, labeling all half-edges (h0–h5), vertices (a, b, c, d), and faces (f0, f1). By systematically writing out every pointer assignment, I was able to get it right without bugs.

Part 5: Edge split

The edge split operation inserts a new vertex at the midpoint of a shared edge, dividing the two adjacent triangles into four. Given two triangles \( (a, b, c) \) and \( (c, b, d) \) sharing edge \( (b, c) \), splitting creates a new vertex \( m \) at the midpoint of \( (b, c) \), and produces four triangles: \( (b, m, a) \), \( (m, c, a) \), \( (c, m, d) \), and \( (m, b, d) \).

My implementation creates 1 new vertex, 3 new edges (\( m\text{-}c \), \( m\text{-}a \), \( m\text{-}d \)), 2 new faces, and 6 new half-edges. I then carefully reassign all pointers using setNeighbors() for all 12 half-edges, and update vertex, face, and edge halfedge() pointers. I also set the isNew flags on edges and vertices to support Loop subdivision later: the two edges connecting \( m \) to the opposite vertices (\( a \) and \( d \)) are marked as new, while the split halves of the original edge are marked as old.

Debugging journey: The most challenging part was getting all 12 half-edge pointer assignments correct. I drew the before/after diagrams on paper, labeling every half-edge (h0–h11), vertex, edge, and face. I found it very helpful to physically draw out all the mesh elements and trace the next pointer of each half-edge to verify correctness. This systematic approach helped me avoid subtle pointer bugs.

Extra Credit: Boundary Edge Split

I also implemented support for splitting boundary edges. In a half-edge mesh, boundary edges have one interior face and one virtual boundary face (a polygon representing the hole in the mesh). The key difference from a regular split is that we only split the interior face into two triangles, while the boundary face simply gains an additional edge segment along its boundary loop.

Specifically, when splitting a boundary edge \( (b, c) \) with interior face \( (b, c, a) \):

  • A new vertex \( m \) is created at the midpoint of \( (b, c) \).
  • The interior triangle is split into two: \( (b, m, a) \) and \( (m, c, a) \).
  • On the boundary side, the half-edge \( c \to b \) is replaced by \( c \to m \to b \), inserting \( m \) into the boundary loop without splitting the virtual boundary face.
  • New elements created: 1 vertex, 2 edges, 1 face, 4 half-edges (compared to 3 edges, 2 faces, 6 half-edges for an interior split).

Part 6: Loop subdivision for mesh upsampling

Loop subdivision is a mesh upsampling algorithm that increases the resolution of a triangle mesh while smoothing its geometry. Our implementation follows a 5-step process:

  1. Compute new positions for old vertices: For each existing vertex, we compute its updated position using the Loop subdivision weighting rule. For an interior vertex with degree \( n \), we use the weight \( u = \frac{3}{16} \) if \( n = 3 \), otherwise \( u = \frac{3}{8n} \). The new position is \( (1 - nu) \cdot v + u \cdot \sum_{\text{neighbors}} v_i \).
  2. Compute new edge midpoint positions: For each original edge, we compute where the new midpoint vertex should be placed. For an interior edge shared by vertices \( A, B \) with opposite vertices \( C, D \), the new position is \( \frac{3}{8}(A + B) + \frac{1}{8}(C + D) \).
  3. Split every original edge: We iterate through all original edges and split each one, creating a new midpoint vertex. The new vertex is assigned the precomputed edge midpoint position and marked as isNew = true.
  4. Flip new edges connecting old and new vertices: Any newly created edge that connects one old vertex to one new vertex is flipped. This restores the correct mesh connectivity for Loop subdivision.
  5. Update all vertex positions: Copy the precomputed newPosition into the final position for all vertices.

A key implementation detail is correctly tracking which edges and vertices are "new" vs "old." We use the isNew flag on both edges and vertices. When splitting, we count the original edges beforehand and only split that many, avoiding infinite loops from splitting newly created edges.

Asymmetry in Cube Subdivision and Pre-processing

When we apply Loop subdivision to a cube, the resulting shape becomes asymmetric. This happens because each face of the original cube has only one diagonal edge, and this diagonal is oriented differently across faces. Since Loop subdivision weights are based on the mesh connectivity, this asymmetric triangulation causes the smoothed result to favor certain directions, producing a lopsided shape instead of a sphere.

To fix this, we can pre-process the cube by splitting the diagonal edge on each face. This turns each face from 2 triangles into 4 symmetric triangles, making the triangulation symmetric across all faces. After pre-processing, the Loop subdivision produces a much more symmetric, sphere-like result.

Extra Credit: Boundary Loop Subdivision

We extended the Loop subdivision algorithm to correctly handle meshes with boundaries. Without special treatment, boundary edges and vertices would be computed using the standard interior rules, leading to incorrect results or crashes.

Our implementation adds the following special rules for boundary elements:

  • Boundary vertex update: For a boundary vertex \( v \) with two neighboring boundary vertices \( a \) and \( b \), the new position is \( \frac{1}{8}(a + b) + \frac{3}{4} v \). This keeps boundary vertices close to their original positions while slightly smoothing along the boundary curve.
  • Boundary edge midpoint: For a boundary edge with endpoints \( A \) and \( B \), the new midpoint vertex is simply placed at the midpoint \( \frac{1}{2}(A + B) \), rather than using the 3/8-1/8 interior rule (since there are no opposite vertices).
  • No flipping of boundary edges: During the edge flip step, we skip any boundary edges to preserve the boundary topology.
